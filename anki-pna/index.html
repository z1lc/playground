🔓 Anki PNA — Local Vulnerability Check

Tests whether your machine is exposed to the localhost-server attack class Tavis Ormandy described (thread). Run only against your own machine.

How this works

When Anki is running, it starts an HTTP server on a localhost port to serve media to its embedded WebView. If that server has permissive CORS and a path-traversal flaw, any website you visit can use your browser to read files from your home directory. This page simulates the malicious-site role: it scans for the server, fingerprints it, then attempts to read ~/.zshrc via path traversal.

For this to be a real cross-origin test, serve this page from a non-Anki origin (e.g. python3 -m http.server 8000 in the playground root and open http://127.0.0.1:8000/anki-pna/). Firefox is the most permissive; Chrome may apply Private Network Access checks.

Environment

Step 1 — Scan localhost for Anki services

Probes :8765 for AnkiConnect (third-party addon) and a range of dynamic ports for Anki's built-in mediasrv. The scan uses fetch with short timeouts to detect listeners. Total scan time: ~10–20s.

AnkiConnect (port 8765)

Step 2 — File read against built-in mediasrv

Path-traversal attempt: the request joins your input onto the server's media base directory. Each .. segment is sent as %2e%2e so the URL parser doesn't collapse it before transmission — the server decodes it back to .. on the other side.

override scan result; useful when fingerprint was CORS-blocked

Mitigation if exposed

      
    
        
  • Update Anki to the latest version. Tavis disclosed this to the developers; expect a patched release.
    • 
        
  • Quit Anki when not actively using it — the server only runs while Anki is open.
    • 
        
  • For AnkiConnect specifically: ensure an apiKey is set in its config so destructive actions are blocked.
    • 
        
  • Use a browser that enforces Private Network Access (recent Chrome). Firefox/Safari don't yet.
